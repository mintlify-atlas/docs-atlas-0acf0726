## Overview

Kairo's risk detection system automatically identifies students who need additional support by analyzing performance metrics, engagement patterns, and soft skills data. Team Leaders receive real-time alerts with actionable insights.

<Note>
  Risk flags are **continuously evaluated** and updated as new data arrives from Moodle progress tracking and assessment results.
</Note>

## Risk Factors

The system evaluates multiple dimensions to determine risk level:

### Academic Performance

<Accordion title="Average Score Thresholds">
  - **Critical:** < 50% average across modules
  - **High:** 50-60% average
  - **Medium:** 60-70% average
  - **Low:** > 70% average
  
  Scores are calculated from `moodle_progress.average_score`.
</Accordion>

<Accordion title="Struggling Topics">
  Students who consistently score below 60% on specific topics across multiple attempts are flagged. The system tracks:
  
  ```javascript
  {
    "struggling_topics": ["SQL Joins", "Normalization", "Indexing"],
    "topic_scores": {
      "SQL Joins": [45, 52, 48],
      "Normalization": [38, 42, 51]
    }
  }
  ```
  
  **Source:** `moodle_progress.struggling_topics` (array of topic IDs)
</Accordion>

<Accordion title="Week-over-Week Decline">
  A drop of 15+ points between consecutive weeks triggers a risk flag:
  
  ```javascript
  if (currentWeekScore - previousWeekScore < -15) {
    createRiskFlag(coderId, 'high', 'Significant score decline');
  }
  ```
</Accordion>

### Engagement Patterns

<Accordion title="Low Attendance">
  Students who miss 3+ consecutive sessions or have < 70% attendance rate over 2 weeks are flagged as high risk.
  
  **Note:** Attendance tracking integration depends on Moodle sync configuration.
</Accordion>

<Accordion title="Assignment Completion Rate">
  Completion rate < 60% for required assignments triggers medium risk:
  
  ```sql
  SELECT 
    coder_id,
    COUNT(*) FILTER (WHERE completed = true) / COUNT(*)::float AS completion_rate
  FROM assignments
  WHERE required = true
  GROUP BY coder_id
  HAVING COUNT(*) FILTER (WHERE completed = true) / COUNT(*)::float < 0.6
  ```
</Accordion>

<Accordion title="Days Since Last Activity">
  - **Medium risk:** 5+ days inactive
  - **High risk:** 10+ days inactive
  - **Critical risk:** 15+ days inactive
</Accordion>

### Soft Skills Indicators

<Accordion title="Low Autonomy + Poor Time Management">
  Students with both `autonomy < 2` and `time_management < 2` require structured support:
  
  ```javascript
  if (softSkills.autonomy < 2 && softSkills.time_management < 2) {
    if (averageScore < 60) {
      createRiskFlag(coderId, 'high', 'Low self-regulation + failing scores');
    } else {
      createRiskFlag(coderId, 'medium', 'Low self-regulation skills');
    }
  }
  ```
</Accordion>

<Accordion title="Communication + Teamwork < 2">
  Indicates isolation risk, especially in collaborative bootcamp environments. May struggle with pair programming and group projects.
</Accordion>

### Risk Flags in Dashboard Data

Risk flags are included in the main dashboard query:

```sql
SELECT 
  u.id, u.full_name,
  mp.average_score, mp.current_week,
  rf.risk_level
FROM users u
LEFT JOIN moodle_progress mp ON u.id = mp.coder_id
LEFT JOIN risk_flags rf ON u.id = rf.coder_id AND rf.resolved = false
WHERE u.role = 'coder' AND u.clan = $1
ORDER BY
  CASE rf.risk_level
    WHEN 'critical' THEN 1
    WHEN 'high'     THEN 2
    WHEN 'medium'   THEN 3
    ELSE 4
  END,
  u.full_name ASC
```

**Note:** Only **unresolved** flags (`resolved = false`) are shown in dashboards.

## Data Model

```sql
CREATE TABLE risk_flags (
  id              SERIAL PRIMARY KEY,
  coder_id        INT NOT NULL REFERENCES users(id),
  risk_level      VARCHAR(20) NOT NULL CHECK (risk_level IN ('low', 'medium', 'high', 'critical')),
  reason          TEXT NOT NULL,
  created_at      TIMESTAMP DEFAULT NOW(),
  resolved        BOOLEAN DEFAULT false,
  resolved_at     TIMESTAMP,
  resolved_by     INT REFERENCES users(id),
  resolution_note TEXT
);

CREATE INDEX idx_risk_flags_coder ON risk_flags(coder_id);
CREATE INDEX idx_risk_flags_unresolved ON risk_flags(coder_id, resolved) WHERE resolved = false;
CREATE INDEX idx_risk_flags_level ON risk_flags(risk_level) WHERE resolved = false;
```

## Automated Risk Detection Workflow

<Steps>
  <Step title="Moodle Data Sync">
    Progress data is synced from Moodle at regular intervals (hourly or on-demand via webhook).
    
    **Updated fields:**
    - `average_score`
    - `current_week`
    - `struggling_topics`
    - `weeks_completed`
  </Step>

  <Step title="Risk Evaluation Trigger">
    After each sync, a background job evaluates all active coders:
    
    ```javascript
    // Pseudo-code for risk evaluation
    for (const coder of activeCoders) {
      const riskFactors = evaluateRiskFactors(coder);
      const riskLevel = calculateRiskLevel(riskFactors);
      
      if (riskLevel !== 'low') {
        upsertRiskFlag(coder.id, riskLevel, riskFactors.reason);
      } else {
        resolveRiskFlag(coder.id, 'Performance improved');
      }
    }
    ```
  </Step>

  <Step title="Risk Flag Creation/Update">
    If a new risk is detected, the system:
    1. Creates a risk flag with level and reason
    2. Sends notification to assigned Team Leader
    3. Logs the event for analytics
    
    If risk improves, existing flags are marked as resolved.
  </Step>

  <Step title="Team Leader Notification">
    Team Leaders receive real-time notifications via the notification service:
    
    ```javascript
    await notifyUser(
      tlId,
      'High Risk Student Alert',
      `${coder.full_name} has been flagged as high risk: ${reason}`,
      'risk_alert',
      riskFlagId
    );
    ```
  </Step>
</Steps>

## Analytics & Reporting

### Risk Trends Query

```sql
-- Weekly risk flag creation trends
SELECT 
  DATE_TRUNC('week', created_at) AS week_start,
  risk_level,
  COUNT(*) AS flags_created,
  COUNT(*) FILTER (WHERE resolved = true) AS resolved_count,
  AVG(EXTRACT(EPOCH FROM (resolved_at - created_at))/3600)::int AS avg_resolution_hours
FROM risk_flags
WHERE created_at > NOW() - INTERVAL '3 months'
GROUP BY week_start, risk_level
ORDER BY week_start DESC, risk_level;
```

### Clan Risk Comparison

```sql
-- Compare risk rates across clans
SELECT 
  u.clan,
  COUNT(DISTINCT u.id) AS total_coders,
  COUNT(DISTINCT rf.coder_id) AS coders_with_risk,
  (COUNT(DISTINCT rf.coder_id)::float / COUNT(DISTINCT u.id) * 100)::numeric(5,2) AS risk_percentage
FROM users u
LEFT JOIN risk_flags rf ON u.id = rf.coder_id AND rf.resolved = false
WHERE u.role = 'coder'
GROUP BY u.clan
ORDER BY risk_percentage DESC;
```

## Performance Considerations

<Warning>
  Risk evaluation can be expensive for large cohorts. Run detection jobs during off-peak hours and use database indexes aggressively.
</Warning>

### Recommended Indexes

```sql
-- Users table
CREATE INDEX idx_users_role_clan ON users(role, clan) WHERE role = 'coder';

-- Risk flags table  
CREATE INDEX idx_risk_flags_unresolved ON risk_flags(coder_id, resolved) WHERE resolved = false;
CREATE INDEX idx_risk_flags_created ON risk_flags(created_at DESC);

-- Moodle progress table
CREATE INDEX idx_moodle_progress_coder ON moodle_progress(coder_id);
CREATE INDEX idx_moodle_progress_score ON moodle_progress(average_score) WHERE average_score < 70;
```
